# 首页悬浮效果规范

这份文档只总结当前首页已经落地、并且用户体验相对稳定的 hover（悬浮态）实现。目的不是追求“所有块都能悬浮”，而是沉淀一套当前项目里已经验证过、比较克制的交互风格，后续详情页、个人页、发帖页如果要补 hover，优先参考这里。

## 1. 先说结论

- 首页 hover 的核心风格是：轻微上浮 + 轻微换色 + 轻微阴影增强
- 交互幅度整体比较小，避免页面看起来发飘
- 不是所有块都要加 hover，只有“明确可点击”的元素才优先考虑
- 非交互型内容块默认不加 hover，尤其不要整页所有卡片一起悬浮

## 2. 当前首页实际在用的 hover 类型

### 2.1 轻按钮型

代表选择器：

- `.forum-nav-link`
- `.forum-pill-button`
- `.forum-tab`
- `.forum-filter`

共同特点：

- 位移通常是 `translateY(-1px)`
- 主要变化是背景色和文字颜色
- 阴影通常不明显增强，重点是“按钮被激活”的轻反馈
- 过渡时间大多在 `200ms` 左右

适用场景：

- 返回按钮
- 轻操作按钮
- 筛选项
- 切换 tab（页签）

不适用场景：

- 大体积内容卡片
- 需要强层次感的帖子列表项

### 2.2 内容卡片型

代表选择器：

- `.forum-post-card`
- `.auth-card`
- `.composer-topic-card`

共同特点：

- 位移通常是 `translateY(-2px)` 到 `translateY(-3px)`
- 阴影会增强，但不会夸张到像弹窗飘起来
- 边框颜色会略微提亮
- 过渡时间大多在 `180ms` 到 `220ms`

首页帖子卡片的典型模式：

```css
transition:
  transform 220ms ease,
  box-shadow 220ms ease,
  border-color 220ms ease,
  background 220ms ease;

.forum-post-card:hover {
  transform: translateY(-3px);
  border-color: rgba(255, 255, 255, 0.98);
  box-shadow:
    inset 0 1px 0 rgba(255, 255, 255, 0.78),
    0 34px 100px rgba(92, 118, 168, 0.22);
}
```

适用场景：

- 整张帖子卡片可点击
- 分类卡片
- 登录卡这种整体可感知为一个操作容器的块

不适用场景：

- 详情页里单纯承载阅读内容的大段正文块
- 评论区里只负责展示信息、没有明显点击行为的嵌套块
- 页面中已经很多层级阴影时，再额外全量 hover 会让视觉变乱

## 3. 首页 hover 的强度边界

当前首页能看出几个比较明确的边界：

- 不会出现 `translateY(-4px)`、`translateY(-6px)` 这种明显跳动
- 不会给所有次级块都加 hover
- 不会让内层块和外层块同时一起浮起来
- 不会一口气同时改位移、缩放、模糊、发光
- 禁用态控件不应该保留 hover 反馈

换句话说，首页的做法偏工程化，不是炫技式交互。

## 4. 后续新增 hover 时的推荐顺序

如果后面要给新页面补 hover，建议按下面顺序判断：

1. 先判断这个块是不是“明确可点击”
2. 如果只是阅读容器，默认不要加 hover
3. 如果是轻按钮，优先用按钮型 hover
4. 如果是整卡点击，优先复用帖子卡片型 hover
5. 一个区域里只保留 1 个主 hover 焦点，避免父子块同时悬浮

## 5. 详情页的落地建议

详情页后续如果要补 hover，建议这样做：

- 顶部返回按钮、操作按钮：直接复用 `.forum-nav-link` / `.forum-pill-button` 风格
- 相关讨论卡片：如果整卡可点击，可以参考 `.forum-post-card` 的 hover，但位移最多 `-2px`
- 正文主卡、评论容器主卡：默认不加 hover
- 评论内部回复块、说明块：默认不加 hover
- 禁用按钮，例如“发送回复”这种占位按钮：明确禁止 hover 和位移反馈

这样做的好处：

- 首页和详情页的交互语言一致
- 页面不会出现“所有东西都在飘”的问题
- 后续新人改样式时，也更容易判断该不该加 hover
