# 前端开发规范

本文档用于当前 `mono` 认证前端项目的后续开发。规范参考了官方文档与当前主流工程实践，优先考虑中小团队易维护、能落地的方案。

## 1. 技术栈基线

- 构建工具使用 `Vite`（前端构建工具）
- 框架使用 `React`（用于构建界面的前端库） + `TypeScript`（带类型系统的 JavaScript 超集）
- 代码检查使用 `ESLint`（静态代码检查工具）
- 本项目当前优先保留最少依赖，减少认证页阶段的维护成本

为什么这样定：

- `Vite` 启动快、热更新快，适合现在这种页面快速联调场景
- `TypeScript` 能把接口字段、组件参数、状态结构约束住，减少联调时低级错误
- 当前团队规模不大时，依赖越少，排障越简单

## 2. 目录约定

- `src/api`：只放接口请求函数，不放页面业务拼装逻辑
- `src/components`：可复用组件，例如提示条、弹窗、表单块
- `src/config`：环境变量、运行时配置
- `src/lib`：纯工具层，例如请求封装、格式化函数
- `src/styles`：全局样式、主题变量
- `src/types`：接口类型、公共类型

要求：

- 页面层尽量只负责组合，不要把请求细节散落在 JSX（React 组件模板语法）里
- 类型定义靠近领域放置，避免一个超大的 `types.ts`

## 3. 接口联调规范

- 所有接口统一走 `src/api`
- 所有请求统一走 `src/lib/http.ts`
- 后端返回统一按 `{ code, msg, data }` 解析
- 联调阶段优先兼容真实后端现状，不要为了“理论更优”先做复杂封装

建议：

- 一个接口一个函数，函数名直接体现业务语义，例如 `login`、`register`
- 页面中不直接拼 URL（统一资源定位符），统一从配置或请求层取
- 登录态、刷新令牌、后续鉴权拦截逻辑，等受保护页面真正接入时再集中补

## 4. 组件开发规范

- 一个组件只解决一个清晰职责
- 组件默认先写成函数组件
- props（组件入参）必须写类型
- 复用价值不高的 UI（界面）结构，不要过度抽成组件

现实建议：

- 中小项目最常见的问题不是“组件抽得不够”，而是“抽太早、抽太散”
- 认证页、设置页、表单页这类业务 UI，先以页面内聚合为主，稳定后再抽

## 5. 状态管理规范

- 页面级状态优先使用 `useState`（React 状态钩子）
- 简单副作用优先使用 `useEffect`（React 副作用钩子）
- 当前阶段不引入全局状态库

为什么：

- 现在只有注册、登录页面，全局状态库会增加理解成本
- 等后面出现菜单权限、用户资料、跨页面缓存，再评估是否引入 `Zustand`（轻量状态库）或其他方案

## 6. 样式规范

- 全局主题变量统一写在 `:root`
- 颜色、圆角、阴影优先走 CSS（层叠样式表）变量
- 样式类名保持语义化，例如 `auth-card`、`feature-card`
- 先保证桌面端还原度，再保证移动端可用性

要求：

- 不要在 JSX 里堆大段行内样式
- 不要把颜色值、间距值四处硬编码
- 复杂视觉效果优先沉淀为变量或块级样式

## 7. 表单与校验规范

- 当前简单表单优先使用受控组件
- 校验逻辑直接靠近页面放置
- 优先做“够用”的同步校验，例如空值、长度、格式

后续扩展建议：

- 如果表单字段明显增多，再引入 `react-hook-form`（React 表单库）和 `zod`（TypeScript 校验库）
- 现在不要为了两个字段把表单体系做得太重

## 8. 提交与错误处理规范

- 提交按钮在请求中必须进入禁用态
- 成功后给明确反馈
- 失败信息优先展示后端返回的 `msg`
- 网络错误和服务错误要区分提示

落地建议：

- 先统一用顶部 `toast`（轻提示）反馈
- 真正接入业务页面后，再按页面重要性决定是否增加模态框或错误页

## 9. 环境变量规范

- 所有前端环境变量统一以 `VITE_` 开头
- `.env.example` 必须保留一份可参考模板
- 不要把真实密钥提交进仓库

## 10. 提交前检查

每次提交前至少做这几件事：

1. `npm run build`
2. 手工点一遍注册、登录主流程
3. 检查移动端布局是否明显错位
4. 检查接口字段名是否和后端文档一致

## 12. 当前项目特别说明

- 现阶段先把认证页打通，不要过早引入复杂路由守卫和权限系统
- 如果后端接口路径后续统一成 `/api/v1/account`，前端只需要改环境变量，不需要大改页面
- 如果后端最终把 `refresh_token` 改成标准 `HttpOnly Cookie`（仅服务端可读 Cookie），前端只保留 `credentials: 'include'` 即可，不要再在页面层处理刷新令牌
